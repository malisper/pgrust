 # Phase 3 Split: 3A B-tree Storage Core, 3B First
  Usable Indexes

  ## Summary

  Split Phase 3 into two implementation slices:

  - Phase 3A: exact PostgreSQL-shaped B-tree disk
    layout and internal methods only. No planner/
    executor/DML use yet; indexes remain not ready/
    not valid.
  - Phase 3B: first usable index support for CREATE
    INDEX, SELECT, INSERT, and COPY FROM, using the
    Phase 3A core. This is the first phase that
    exposes indexes to query execution.

  This keeps the storage/algorithm port separate
  from SQL-visible wiring.

  ## Phase 3A

  - Replace the single src/backend/access/nbtree/
    mod.rs stub with PostgreSQL-shaped modules:
    nbtree, nbtpage, nbtsearch, nbtinsert,
    nbtcompare, nbtpreprocesskeys, nbtsplitloc,
    nbtutils, nbtsort.
    Keep nbtdedup, nbtvalidate, and nbtxlog as
    file-layout placeholders if they are not fully
    used yet.
  - Add the missing shared access definitions in
    src/include/access/nbtree.rs and expand the
    existing index access headers so the B-tree
    code can use PostgreSQL-shaped constants and
    structs.
  - Implement the exact on-disk B-tree page format:
      - metapage layout and version fields
      - BTPageOpaqueData special space
      - page flags, sibling links, level, cycle ID
        fields
      - leaf tuples, pivot tuples, high keys,
        downlinks, heap-TID tiebreak semantics
      - posting-list-capable tuple encoding/
        decoding, even if dedup itself is not
        turned on yet
  - Implement internal B-tree methods, but do not
    expose them through SQL yet:
      - page init/verification
      - binary search within a page
      - scan-key preprocessing and tuple comparison
      - descent stack/search helpers
      - page-local insert positioning
      - split-point selection and page split
        primitives
      - root/metapage bootstrap helpers
      - internal scan-position structs for later
        amgettuple
  - Add an in-memory build spool/sort helper in
    nbtsort for later use by btbuild.
    Phase 3A does not need external tape sort or
    parallel build.
  - Keep current Phase 2 behavior for SQL:
      - CREATE INDEX may still create catalog
        metadata
      - built indexes remain indisvalid = false and
        indisready = false
      - planner/executor/DML still ignore them

  ## Phase 3B

  - Wire the B-tree core into the generic AM
    callbacks:
      - ambuildempty
      - ambuild
      - aminsert
      - ambeginscan
      - amrescan
      - amgettuple
      - amendscan
  - Make non-concurrent CREATE INDEX on permanent
    tables actually build the index from existing
    heap rows using the Phase 3A in-memory sorted-
    build path.
    On success, set indisvalid = true and
    indisready = true.
    If the build would exceed the configured in-
    memory build budget, return a clear unsupported
    error.
  - Add the first execution-visible index plan/
    runtime node:
      - Plan::IndexScan
      - IndexScanState
        Keep IndexOnlyScan, BitmapIndexScan, and
        BitmapHeapScan out of scope for this phase.
  - Add deterministic planner rules for base-table
    scans only:
      - an index is usable only if a leading prefix
        of index keys helps
      - equality quals consume leading columns
        first
      - then at most one range column may be used
      - ORDER BY can use the index only when it
        matches the remaining leading index order
      - longest usable prefix wins, then ORDER BY
        satisfaction, then catalog order
      - any leftover predicates remain as a normal
        Filter above the IndexScan
  - Support read/write behavior only for:
      - SELECT
      - INSERT
      - COPY FROM
      - EXPLAIN
  - Maintain indexes for INSERT and COPY FROM.
    Until later phases, reject UPDATE, DELETE, and
    TRUNCATE on tables that have any valid index,
    since delete/update maintenance is not
    implemented yet.
  - Keep these out of scope for Phase 3B:
      - unique enforcement
      - partial/expression/include indexes
      - temp-table indexes
      - concurrent index builds
      - page deletion / VACUUM index cleanup
      - WAL / recovery for index operations
      - index-only scans
      - bitmap scans

  ## Test Plan

  - Phase 3A:
      - unit tests for metapage, opaque-area, high-
        key, pivot-tuple, and posting-list-capable
        tuple encoding/decoding
      - split/search helper tests on synthetic
        pages
      - deterministic empty-index relfile checks
        against PostgreSQL 18.3 page layout
  - Phase 3B:
      - CREATE INDEX on a populated permanent table
        produces a usable index and marks it ready/
        valid
      - equality, prefix-range, and ORDER BY
        queries choose IndexScan and return the
        same rows as seqscan
      - backward ordered scans work when planner
        selects them
      - INSERT and COPY FROM maintain the index
        correctly
      - UPDATE, DELETE, and TRUNCATE on indexed
        tables raise the planned unsupported error
      - builds that exceed the in-memory build
        budget fail with the planned unsupported
        error
      - EXPLAIN shows IndexScan when selected
  - Cross-check deterministic small builds against
    PostgreSQL 18.3 relfiles where page contents
    should match exactly.

  ## Assumptions

  - PostgreSQL 18.3 in ../postgres remains the
    canonical source for file layout, page format,
    and internal algorithm structure.
  - Phase 3 is now explicitly split into 3A and 3B;
    later phases handle delete/update maintenance,
    uniqueness, VACUUM/page deletion, WAL/recovery,
    bitmap scans, and index-only scans.
  - The first build path is in-memory only. No
    external sort/tape machinery or parallel build
    is required in this split phase.
  - Since pgrust does not yet have visibility-map-
    driven index-only infrastructure, Phase 3B
    exposes only ordinary IndexScan.
